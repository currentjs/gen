# @currentjs/gen - LLM Context Documentation

> **Purpose**: This document provides context for AI/LLM assistants when making code changes to the `@currentjs/gen` package or applications generated by it.

## 1. What Is This?

`@currentjs/gen` is a CLI code generator that transforms YAML specifications into fully functional TypeScript applications following Clean Architecture principles. It generates:

- **Domain Layer**: Entities, Value Objects, Aggregates
- **Application Layer**: Use Cases, Services, DTOs
- **Infrastructure Layer**: Controllers (API + Web), Stores (Database Access)
- **Presentation Layer**: HTML Templates

The tool uses a single YAML configuration format with `domain`, `useCases`, `api`, and `web` sections following Clean Architecture patterns.

## 3. Module YAML Configuration

The new YAML format follows Clean Architecture with four main sections:

### 3.1 Domain Layer (`domain`)

Defines aggregates (entities) and value objects.

```yaml
domain:
  aggregates:
    Invoice:
      root: true                          # Marks as aggregate root
      fields:
        number: { type: string }
        date: { type: datetime }
        amount: { type: Money }           # Value object (define in valueObjects)
      entities:
        - InvoiceItem                     # Child entities
      
    InvoiceItem:
      fields:
        productId: { type: id }
        quantity: { type: integer }
        price: { type: Money }

  valueObjects:
    Money:
      fields:
        amount: { type: decimal, constraints: { min: 0 } }
        currency: { type: enum, values: ['USD', 'EUR', 'PLN'] }
```

**Field Types**: `string`, `number`, `integer`, `decimal`, `boolean`, `datetime`, `date`, `id`, `json`, `array`, `object`, `enum`

**Constraints**: `min`, `max`, `pattern`, `unique`

**Auto-generated Fields**: `id`, `owner_id`, `created_at`, `updated_at`, `deleted_at`

### 3.2 Use Case Layer (`useCases`)

Defines use cases with input/output specifications and handler chains.

```yaml
useCases:
  Invoice:                                # Per aggregate
    create:
      input:
        from: Invoice                     # Take fields from aggregate
        pick: [number, date, amount]      # Only these fields
        # omit: [status]                  # Alternative: exclude fields
        add:                              # Additional input fields
          customerId: { type: id, required: true }
          items: 
            type: array
            of: InvoiceItem
            pick: [productId, quantity]
            optional: true
        validate:
          number: { pattern: "^INV-\\d+$" }
          amount: { min: 0 }
      output:
        from: Invoice
        include:
          items: 
            from: InvoiceItem
            pick: [id, productId, quantity]
      handlers:
        - validateInput                   # Custom → InvoiceService.validateInput(result, input)
        - default:create                  # Built-in → InvoiceService.create(input)
        - notifyAccounting                # Custom → InvoiceService.notifyAccounting(result, input)

    list:
      input:
        pagination:
          type: cursor                    # or: offset
          defaults:
            limit: 20
            maxLimit: 100
        filters:
          status: 
            type: string
            enum: [draft, pending, paid, overdue]
            optional: true
          dateFrom: { type: datetime, optional: true }
          dateTo: { type: datetime, optional: true }
          search: 
            type: string
            searchIn: [number, customerName]
            optional: true
        sorting:
          allow: [date, amount, number]
          default: { field: date, order: desc }
      output:
        from: Invoice
        pick: [id, number, date, amount, status]
        add:
          customerName: { type: string, source: customer.name }
        pagination: true                  # Wraps in { items, cursor/page, total }
      handlers:
        - default:list

    get:
      input:
        identifier: id
      output:
        from: Invoice
        include:
          items:
            from: InvoiceItem
          customer:
            from: Customer
            pick: [id, name, email]
      handlers:
        - default:get
        - loadItems                       # Custom handler

    update:
      input:
        identifier: id
        from: Invoice
        partial: true                     # All fields become optional
        pick: [number, date, amount, status]
      output:
        from: Invoice
      handlers:
        - default:update
        - recalculateTotal

    delete:
      input:
        identifier: id
      output: void
      handlers:
        - default:delete
```

**Handler Types**:
- `default:list` → Service method `list(page, limit)`
- `default:get` → Service method `get(id)`
- `default:create` → Service method `create(input)`
- `default:update` → Service method `update(id, input)`
- `default:delete` → Service method `delete(id)`
- `customName` → Service method `customName(result, input)` (empty implementation for user to fill)

### 3.3 API Layer (`api`)

Defines REST API endpoints.

```yaml
api:
  Invoice:
    prefix: /api/invoice
    endpoints:
      # Public access
      - method: GET
        path: /
        useCase: Invoice:list
        auth: all
        
      # Any authenticated user
      - method: GET
        path: /:id
        useCase: Invoice:get
        auth: authenticated
        
      # Authenticated user can create
      - method: POST
        path: /
        useCase: Invoice:create
        auth: authenticated
        
      # Owner OR admin can update (array syntax)
      - method: PUT
        path: /:id
        useCase: Invoice:update
        auth: [owner, admin]
        
      # Only admin role
      - method: DELETE
        path: /:id
        useCase: Invoice:delete
        auth: admin
```

**Auth Options**:
- `all`: Public access (no authentication required)
- `authenticated`: Any logged-in user
- `owner`: User must own the resource (checks `ownerId` field after fetching entity)
- `admin`, `editor`, etc.: User must have this specific role
- `[owner, admin]`: Array syntax - user must match ANY of these (OR logic). When `owner` is combined with roles, privileged roles bypass the ownership check.

### 3.4 Web Layer (`web`)

Defines web page routes with views.

```yaml
web:
  Invoice:
    prefix: /invoice
    layout: dashboard                   # Layout template name
    pages:
      # Public list
      - path: /
        useCase: Invoice:list
        view: invoiceList
        auth: all

      # Authenticated can access create form
      - path: /create
        method: GET
        view: invoiceCreate
        auth: authenticated
        
      - path: /create
        method: POST
        useCase: Invoice:create
        auth: authenticated
        onSuccess: 
          redirect: /invoice/:id
          toast: "Invoice created"
        onError:
          stay: true
          toast: error
      
      # Public detail view    
      - path: /:id
        useCase: Invoice:get
        view: invoiceDetail
        auth: all
      
      # Owner or admin can edit
      - path: /:id/edit
        method: GET
        useCase: Invoice:get
        view: invoiceEdit
        auth: [owner, admin]
        
      - path: /:id/edit
        method: POST
        useCase: Invoice:update
        auth: [owner, admin]
        onSuccess:
          back: true
          toast: "Updated"
```

## 5. Command Reference

### `currentjs create app [name]`
**File**: `src/commands/createApp.ts`

Creates a new application with:
- `package.json` with dependencies
- `tsconfig.json` for TypeScript
- `app.yaml` application config
- `src/app.ts` entry point
- `src/system.ts` DI decorator (`@Injectable`)
- `src/common/ui/templates/` with layout templates
- `web/app.js` frontend script
- `web/translations.json` for i18n

### `currentjs create module <name>`
**File**: `src/commands/createModule.ts`

Creates a new module with:
- Module directory structure (Clean Architecture folders)
- `{name.toLowerCase()}.yaml` with CRUD template
- Updates `app.yaml` to include the module

### `currentjs generate [module] [--force] [--skip]`
**File**: `src/commands/generateAll.ts`

Generates all TypeScript files from YAML:
1. Reads `app.yaml` to find modules and providers
2. For each module (with `domain` + `useCases`), generates files for each layer
3. Ensures `src/system.ts` exists (creates it for older apps)
4. Scans generated files for `@Injectable` and `@Controller` decorators
5. Builds dependency graph from constructor parameters, topologically sorts
6. Updates `src/app.ts` with DI-based wiring code between markers
7. Runs `npm run build`

**Flags**:
- `--force`: Overwrite modified files
- `--skip`: Skip modified files without prompting

### `currentjs diff [module]`
**File**: `src/commands/diff.ts`

Shows differences between:
- Current file content
- Freshly generated content (with committed patches applied)

Uses hunk-based diff algorithm for precise change tracking.

### `currentjs commit [files...]`
**File**: `src/commands/commit.ts`

Saves user modifications as reusable "hunks":
- Computes diff between generated and current file
- Stores hunks in `registry.json`
- Creates commit file in `commits/` directory

### `currentjs infer --file <path> [--write]`
**File**: `src/commands/infer.ts`

Reverse-engineers YAML model from existing TypeScript entity:
- Parses TypeScript class constructor
- Extracts field types and requirements
- Outputs YAML snippet (or writes to module YAML with `--write`)

### `currentjs migrate commit`
**File**: `src/commands/migrateCommit.ts`

Creates database migration files from schema changes.

## 6. Generator Files Reference

### `domainLayerGenerator.ts`
Generates entity classes and value objects from `domain` config.

**Key Methods**:
- `generateAggregate()`: Creates entity class with constructor, setters
- `generateValueObject()`: Creates value object with validation

**Generated Features**:
- Constructor with typed parameters
- Setter methods for each field
- Value object validation in constructor
- Enum type generation for value objects

### `dtoGenerator.ts`
Generates Data Transfer Objects from `useCases` config.

**For Each Use Case**:
- `{Model}{Action}Input` class with `parse()` method
- `{Model}{Action}Output` class with `from()` method

**Features**:
- Input validation
- Type coercion (strings to numbers, dates)
- Pagination handling
- Filter/sorting support

### `useCaseGenerator.ts`
Generates use case orchestrators that coordinate handlers. Generated use cases are decorated with `@Injectable` (imported from `src/system.ts`).

**Features**:
- Orchestrates service handler calls in sequence
- For aggregate roots: includes `getResourceOwner(id)` method (delegates to service)
- Used by controllers for pre-mutation authorization

**Generated Code**:
```typescript
export class InvoiceUseCase {
  constructor(private invoiceService: InvoiceService) {}

  async create(input: InvoiceCreateInput): Promise<Invoice> {
    const result0 = await this.invoiceService.validateInput(null, input);
    const result1 = await this.invoiceService.create(input);
    const result = await this.invoiceService.notifyAccounting(result1, input);
    return result;
  }
  
  // For aggregate roots: ownership lookup for authorization
  async getResourceOwner(id: number): Promise<number | null> {
    return await this.invoiceService.getResourceOwner(id);
  }
}
```

### `serviceGenerator.ts`
Generates service classes with handler methods. Generated services are decorated with `@Injectable` (imported from `src/system.ts`).

**Default Handlers**: Full implementation for CRUD operations
**Custom Handlers**: Empty method stubs for user implementation
**Authorization**: `getResourceOwner(id)` method for aggregate roots

```typescript
export class InvoiceService {
  constructor(private invoiceStore: InvoiceStore) {}

  async list(page: number = 1, limit: number = 20) { /* ... */ }
  async get(id: number) { /* ... */ }
  async create(input: any) { /* ... */ }
  async update(id: number, input: any) { /* ... */ }
  async delete(id: number) { /* ... */ }
  
  // For aggregate roots: ownership lookup for authorization
  async getResourceOwner(id: number): Promise<number | null> { /* ... */ }
  
  // Custom handlers (empty for user to implement)
  async validateInput(result: any, input: any): Promise<any> {
    return result;
  }
}
```

### `storeGenerator.ts`
Generates database access layer. Generated stores are decorated with `@Injectable` (imported from `src/system.ts`).

**Features**:
- Row interface (database representation)
- `rowToModel()` conversion with type handling
- `insert()`, `update()`, `getById()`, `getAll()`, `count()`, `softDelete()`
- `getResourceOwner(id)` for aggregate roots (efficient ownership lookup)
- Value object serialization/deserialization
- Datetime string conversion

### `controllerGenerator.ts`
Generates HTTP controllers with access control.

**API Controllers**:
- REST endpoint decorators (`@Get`, `@Post`, etc.)
- Request parsing with DTO
- Use case invocation
- Response transformation
- Pre-mutation owner checks (for update/delete)
- Post-fetch owner checks (for get)

**Web Controllers**:
- `@Render` decorator for templates
- Form handling with strategies
- Success/error handling
- Same access control as API controllers

**Access Control Implementation**:
The generator produces secure auth checks with different timing based on operation type:
1. **Authentication check**: Always runs first to validate user identity and role
2. **Pre-mutation owner check**: For mutations (update/delete), validates ownership BEFORE the operation
3. **Post-fetch owner check**: For reads (get), validates ownership after fetching

```typescript
// Generated code for UPDATE with auth: [owner, admin]
async update(context: IContext): Promise<any> {
  // Step 1: Authentication check
  if (!context.request.user) {
    throw new Error('Authentication required');
  }

  const input = UpdateInput.parse({ ...context.request.body, id: context.request.parameters.id });
  
  // Step 2: PRE-MUTATION owner validation (prevents unauthorized changes)
  const resourceOwnerId = await this.useCase.getResourceOwner(input.id);
  if (resourceOwnerId === null) {
    throw new Error('Resource not found');
  }
  const isOwner = resourceOwnerId === context.request.user?.id;
  const hasPrivilegedRole = context.request.user?.role === 'admin';
  if (!isOwner && !hasPrivilegedRole) {
    throw new Error('Access denied: you do not own this resource');
  }

  // Step 3: Execute the operation (only after authorization passes)
  const result = await this.useCase.update(input);
  return Output.from(result);
}

// Generated code for GET with auth: owner
async get(context: IContext): Promise<any> {
  if (!context.request.user) {
    throw new Error('Authentication required');
  }

  const input = GetInput.parse({ id: context.request.parameters.id });
  const result = await this.useCase.get(input);
  
  // POST-FETCH owner validation (for reads only)
  if (result.ownerId !== context.request.user?.id) {
    throw new Error('Access denied: you do not own this resource');
  }
  
  return Output.from(result);
}
```

Generates HTML templates for web pages.

**Template Types**:
- List templates with table and pagination
- Detail templates with field display
- Create/Edit forms with field inputs
- Form attributes for frontend handling (`data-strategy`, etc.)

## 7. Important Patterns & Conventions

### Access Control System

The generator supports flexible role-based access control with secure owner validation:

**Auth Configuration Types**:
```yaml
# Single values
auth: all           # Public access (no auth)
auth: authenticated # Any logged-in user
auth: owner         # User must own the resource
auth: admin         # User must have 'admin' role

# Array syntax (OR logic)
auth: [owner, admin]      # Owner OR admin
auth: [editor, admin]     # Editor OR admin role
auth: [owner, editor, admin]  # Any of these
```

**How It Works**:

1. **Authentication checks** (before any operation):
   - `all`: No check
   - `authenticated`: Requires `context.request.user` to exist
   - Role names: Requires user + matching role
   - Arrays without `owner`: Requires user + any matching role

2. **Owner validation** (depends on operation type):
   - **For reads (get, list)**: Post-fetch check - compares `result.ownerId` with `context.request.user.id`
   - **For mutations (update, delete)**: Pre-mutation check - calls `getResourceOwner(id)` BEFORE the operation to prevent unauthorized changes

3. **getResourceOwner method** (generated for aggregate roots):
   - Added to Store → Service → UseCase layers
   - Fetches only `ownerId` field for efficient authorization
   - Called by controllers before mutations when `owner` auth is specified

**Entity Requirements for Owner Auth**:
- Aggregate roots automatically have an `ownerId` field (auto-generated)
- The field stores the ID of the user who created the resource
- On create, `ownerId` is injected from `context.request.user.id`

### Dependency Injection System

The generator includes a decorator-driven DI system that auto-wires all module classes in `src/app.ts`.

**Key File**: `src/utils/diResolver.ts`

**How It Works**:
1. Generators decorate Stores, Services, and UseCases with `@Injectable()` (imported from `src/system.ts`). Controllers use the existing `@Controller()` decorator instead.
2. After generation, `generateAll.ts` scans all module `.ts` files for classes with `@Injectable` or `@Controller` decorators.
3. For each discovered class, it parses the constructor to extract dependency types (e.g., `constructor(private invoiceStore: InvoiceStore)`).
4. A dependency graph is built and topologically sorted (Kahn's algorithm) to determine instantiation order.
5. The wiring code (imports + instantiations + controllers array) is injected into `src/app.ts` between `// currentjs:controllers:start` and `// currentjs:controllers:end` markers.

**`@Injectable` Decorator**:
Lives in the generated app's `src/system.ts` (not in any `@currentjs/*` package). It's a simple no-op marker decorator:
```typescript
export function Injectable() {
  return function (target: any) {
    target.__injectable = true;
  };
}
```

**Provider Resolution**:
Database providers are configured in `app.yaml` (global) with optional per-module overrides. The `resolveProviderImport()` function handles both:
- **npm packages**: `@currentjs/provider-mysql` → imported as-is
- **local paths**: `./src/common/SomeProvider` → resolved relative to `src/app.ts`

Stores receive the appropriate provider instance as their first constructor argument based on their module's database configuration.

**`diResolver.ts` Functions**:
- `parseClassFile(filePath)` — Reads a `.ts` file, detects `@Injectable`/`@Controller` decorators, parses constructor params
- `scanModuleClasses(moduleDir)` — Recursively finds all `@Injectable`/`@Controller` classes in a module
- `buildInstantiationOrder(classes, providerVarByType, classProviderVar, srcDir)` — Builds dependency graph and returns topologically sorted instantiation steps
- `resolveProviderImport(importSpec, appTsDir)` — Resolves provider import path for npm or local sources

**Deduplication**: The generator prevents duplicate imports and instantiations when the same class appears across multiple dependency chains.

**Circular Dependencies**: Detected during topological sort; throws an informative error listing the cycle.

### Handler Chain Execution
Handlers execute sequentially. Each custom handler receives:
- `result`: Output from previous handler (or `null` if first)
- `input`: Parsed input DTO

The final result is returned from the last handler.

### Type Mapping
```typescript
// YAML Type → TypeScript Type → Database Type
string → string → VARCHAR
number → number → INT/DECIMAL
boolean → boolean → BOOLEAN
datetime → Date → DATETIME (stored as string in MySQL)
id → number → INT (foreign key)
```

### Value Object Handling
- Domain: Full value object (e.g. `amount: Money` when type is a value object name)
- DTO: Flattened or as value object depending on use case
- Store: Serialized to/from database format

### Registry System
`registry.json` tracks:
- File hashes for change detection
- Committed hunks for reapplication
- Generation timestamps

### Marker Comments
`src/app.ts` uses markers for safe regeneration of DI wiring:
```typescript
// currentjs:controllers:start
import { InvoiceStore } from './modules/Invoice/infrastructure/stores/InvoiceStore';
// ... all imports ...

const db = new MySQLProvider(config.database);
const invoiceStore = new InvoiceStore(db);
const invoiceService = new InvoiceService(invoiceStore);
const invoiceUseCase = new InvoiceUseCase(invoiceService);

const controllers = [
  new InvoiceApiController(invoiceUseCase),
  new InvoiceWebController(invoiceUseCase),
];
// currentjs:controllers:end
```
Everything between these markers is regenerated on each `currentjs generate` run.

## 8. LLM-Specific Instructions

### When Modifying Generators

1. **Update type definitions**: Modify `src/types/configTypes.ts` for new YAML options
2. **Maintain template consistency**: Use existing patterns in `templates/` folder
3. **Update this document**: if there's a mismatch with this document after modifying, you have to update this document respectively

### When Adding New Features

1. **Start with types**: Define interfaces in `configTypes.ts`
2. **Add to generators**: Implement in relevant generator file
3. **Update templates**: Add template snippets if needed
4. **Wire in CLI**: Update `cli.ts` and relevant command files
5. **Test end-to-end**: Create test module and verify generation (if `test-stand` is already created, or the user points out to it, you may just rebuild the package and use the command `generate --force`)
6. **Update this document**: describe new feature in the relative category

### Common Pitfalls

1. **Don't modify generated code directly**: Changes will be overwritten
2. **Use `currentjs commit` for customizations**: Preserves changes across regeneration
3. **Field names are case-sensitive**: `userId` ≠ `UserId`
4. **Handlers execute in order**: First to last in array

### Code Style

- Use `eslint-disable-next-line no-console` for console output
- Use `colors.ts` for terminal styling
- Use `writeGeneratedFile()` for proper hash tracking
- Prefer async/await over callbacks
